# Native PDF Bridge Protocol

Sparrow's final PDF core should be PDF4QT. The current Electron app keeps a
tested bridge ready for that migration while retaining the PyMuPDF backend
fallback.

## Transport

The native host executable is launched as:

```bash
inkwell-pdf4qt-host --stdio-json
```

Requests and responses are newline-delimited JSON objects. Every request has:

- `jsonrpc`: `"2.0"`
- `id`: unique request id generated by Electron
- `method`: one command from `shared/native-pdf-commands.ts`
- `params`: command-specific object

The host returns one line with the same `id` and either `result` or `error`.

## Ownership

- Electron owns process lifecycle, timeouts, and pending request mapping.
- The native host owns PDF4QT document loading, rendering/editing state, and
  undo/redo.
- Renderer ACP tools should call the same commands so manual edits and AI edits
  use one operation model.

## Fallback

If `INKWELL_PDF4QT_HOST` is missing or points to a non-existent executable,
Sparrow continues using pdf.js rendering and PyMuPDF writes. The UI reports this
explicitly in the Native Core panel.
